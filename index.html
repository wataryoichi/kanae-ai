新規開発の中心は、ゼロから構築ではなく接続と計測へ。

kanae.ai は単体のCLI集ではなく、既存のAI/dev資産をつなぐ実行アーキテクチャです。曖昧な依頼を作業オーダーへ変換し、期待ROIに応じてモデルとツールを選び、成果を知識として残します。

Ticket / Intake

音声やテキストの依頼を、目的、受入条件、工数、不明点を含むAI作業オーダーに構造化します。

AI Power Router

hiyu が期待ROI、コスト、機密性、実行条件を見て、ローカルLLM、小型モデル、高性能モデルへ振り分けます。

意図から成果までを、一つの流れにする。

人が話す依頼、AIが使う文脈、実行結果、組織に残す知識を分断しません。各層は既存ツールを活かしながら、運用として測れる形に揃えます。

Intake

曖昧な依頼を、AIがそのまま着手できる作業オーダーへ翻訳。

Route

期待ROIでスコアリングし、最適なコスト帯と実行先へ配分。

Foundry

if-then、FAQ、評価データ、テストを社内知識として蓄積。

# 1. 依頼を作業オーダーに変換
$ ticket-intake "この画面の色を直して、公開前に確認して"

# 2. hiyu がモデルと実行先を選ぶ
$ hiyu route --roi expected --privacy local-first
tier: local + cached context

# 3. kanae が既存ツール群で実行
$ kanae fulfill order.json

# 4. engawa / dashboard へ結果を残す
$ engawa post --from-result result.json

プロダクト群で、AI運用のOSを作る。

kanae.ai はひとつの巨大アプリではありません。受付、ルーティング、実行、文書知識、会話知識を分け、それぞれを連携させてAI運用の流れを作ります。

UDQ

契約書、議事録、規程、PDFを社外に出さず、検索、要約、Q&A、引用つき回答に使えるローカル文書AI基盤です。

engawa

チームの会話をAIが要約、検索、質問応答できる知恵に変える、ビジネスチャット兼ナレッジベースです。

Operating Dashboard

AI ROI、限界AI ROI、余剰枠率、資産化件数を見える化し、AI利用を感覚ではなく運用指標で管理します。

提供は、まだ限定的に。

kanae.ai はフリー配布のユーティリティ集ではありません。現時点ではPoC、個別導入、ライセンス発行を前提に、利用環境、文書管理要件、AI実行ポリシーを確認してから提供します。

PoC Access

評価用ライセンスと対象コンポーネントを個別に案内します。相談窓口は準備中のため、当面は個別連絡ベースで開始します。PoCの確認項目は docs にまとめています。

Licensed Bundle

実行層は署名済みlicenseを起動時に検証します。期限、顧客、feature listを分け、PoCと本番導入を切り替えられます。

Download Gate

download.kanae.ai は将来の限定配布口です。今は一般公開ダウンロードではなく、準備中のプレースホルダーです。

kanae.ai の名前空間。

まずは公式入口として公開し、後からdocs、license、status、限定downloadを分けます。製品群をばらばらに見せず、AI Operating Systemとして束ねます。
